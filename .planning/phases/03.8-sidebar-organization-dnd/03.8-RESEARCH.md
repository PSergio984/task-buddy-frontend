## Summary

This phase focuses on upgrading the Sidebar from a static list to an interactive management hub. Key features include reordering Projects and Tags via Drag-and-Drop (DND) and providing full CRUD capabilities through inline dropdown menus. The system will leverage the existing `@dnd-kit` library for DND and reuse existing modal components for editing.

**Primary recommendation:** Use `@dnd-kit/sortable` with a dedicated vertical grip handle to ensure accessibility and prevent accidental drags while interacting with the sidebar's filtering/CRUD buttons.

## Architecture Patterns

### Recommended Project Structure
```
src/
├── components/
│   ├── sidebar/             # New folder for sidebar sub-components
│   │   ├── sortable-item.tsx # Reusable wrapper for Projects/Tags
│   │   └── grip-handle.tsx   # Dedicated grip handle component
├── hooks/
│   ├── useProjects.ts       # Already contains useReorderProjects
│   └── useTags.ts           # Already contains useReorderTags
```

### Pattern 1: Multi-Context DND
Since Projects and Tags are separate sections, each should have its own `SortableContext` but share a single `DndContext` if they exist within the same Sidebar container. However, per CONTEXT.md, they cannot be dragged across sections, so separate containers are preferred.

### Anti-Patterns to Avoid
- **Implicit Dragging:** Don't allow the whole item to be the drag handle; it interferes with text selection and clicking to filter.
- **Optimistic Sync Gaps:** Ensure the `onMutate` logic in `useReorderProjects` correctly handles the local state update to prevent "flicker" before server confirmation.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Reordering Logic | Array index swapping | `@dnd-kit/sortable` | Handles edge cases, accessibility (ARIA), and touch support. |
| Dropdown Menus | Custom portal/positioning | `DropdownMenu` (Radix/Shadcn) | Handles keyboard navigation and positioning automatically. |

## Common Pitfalls

### Pitfall 1: React 19 Ref Handling
**What goes wrong:** Passing `setNodeRef` directly to an element's `ref` prop might cause TypeScript errors or runtime issues due to React 19's cleanup function support in refs.
**How to avoid:** Use a functional ref: `ref={(node) => { setNodeRef(node); }}` if the standard assignment fails.

### Pitfall 2: Filter Invalidation on Deletion
**What goes wrong:** User deletes the active project; the UI shows a "404 Not Found" or empty state because the filter remains on the deleted ID.
**How to avoid:** Explicitly call `setActiveSidebarFilter("all")` inside the `onSuccess` or before the delete mutation in the component.

## Code Examples

### Sortable Item Structure
```typescript
import { useSortable } from '@dnd-kit/sortable';
import { CSS } from '@dnd-kit/utilities';

export function SortableSidebarItem({ id, children }) {
  const {
    attributes,
    listeners,
    setNodeRef,
    transform,
    transition,
    isDragging
  } = useSortable({ id });

  const style = {
    transform: CSS.Transform.toString(transform),
    transition,
    opacity: isDragging ? 0.5 : 1,
  };

  return (
    <div ref={setNodeRef} style={style} className="group">
      <div {...attributes} {...listeners}>
        <GripVertical className="h-4 w-4" />
      </div>
      {children}
    </div>
  );
}
```
