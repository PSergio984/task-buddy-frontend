# Phase 04.7: Notification System Integration - Context

## Decisions

### 1. Notification Bell UI
- **Location:** `TopNav`, positioned to the left of the User Profile.
- **Component:** A bell icon with a red badge for unread counts. Clicking opens a Scrollable Popover.
- **Functionality:**
  - Display last 20 notifications.
  - "Mark all as read" action.
  - Item click: Mark as read via API and navigate to the task link (`action_url`).
  - Different styles for read/unread items.

### 2. Browser Push Integration
- **Opt-in Flow:** Users must explicitly enable push notifications via a toggle in the **Profile Settings**.
- **VAPID Key:** The backend provides the public key (generated in Phase 3.75).
- **Service Worker:** Ensure `vite-plugin-pwa` is configured to handle push events (even if just for registration).
- **Backend Sync:** Post the `PushSubscription` object to `/api/v1/notifications/push-subscription`.

### 3. State & Update Strategy
- **Polling:** Use TanStack Query `refetchInterval: 120000` (2 minutes) for the notifications list.
- **Focus:** Enable `refetchOnWindowFocus: true` to refresh when the user returns to the tab.
- **Store:** Store the total unread count in a way that the bell badge can access it easily (e.g., `useNotifications` hook).

### 4. Real-time Feedback (Toasts)
- **Trigger:** When a poll reveals a *new* high-priority notification (`REMINDER_DUE`, `REMINDER_OVERDUE`), show a persistent `toast`.
- **Deduplication:** Ensure the same notification doesn't toast twice if the user stays on the same page.

## Locked Implementation Choices
- **UI Components:** Use Radix UI `Popover` (via Shadcn) for the bell dropdown.
- **Toasts:** Use existing `use-toast` hook.
- **API:** Use the established `axios` instance and TanStack Query.

## Gray Areas Resolved
- **Placement:** TopNav (not sidebar).
- **Permission:** Explicit Opt-in (not automatic).
- **Toast Scope:** High-priority only.
